# 개발 플로우 규칙 (SSOT)

이 문서는 개발 프로세스와 작업 원칙을 정의한다.

## 작업 원칙

### THIS TASK ONLY
- **한 TASK = 한 목적**
- 작업 범위를 명확히 정의
- 작업 종료 시 변경 파일 명시

### SSOT 우선 원칙
- 코드 변경 전 SSOT 문서 확인
- SSOT에 없는 결정은 구현 금지
- 변경 시 SSOT 선반영

## 개발 플로우

### 1. 작업 시작
1. 작업 문서 확인 (`docs/dev_task/` 디렉토리)
2. SSOT 문서 확인 (`docs/_ssot/`)
3. 작업 범위 명확히 정의

### 2. 개발 중
1. SSOT 규칙 준수
2. 코드 리뷰 기준 준수
3. 테스트 작성 (가능한 경우)

### 3. 작업 완료
1. 변경 파일 명시
2. SSOT 문서 업데이트 (필요 시)
3. 체크리스트 확인

## 코드 변경 원칙

### 금지 사항
- 비즈니스 로직 변경 (작업 범위 외)
- DB 스키마 변경 (작업 범위 외)
- 기존 API 동작 변경 (작업 범위 외)

### 허용 사항
- 코드 구조 개선
- 문서화
- TODO 주석 추가
- Skeleton/Stub 생성

## 작업 종료 체크리스트
- [ ] 작업 목적 달성 확인
- [ ] 변경 파일 목록 정리
- [ ] SSOT 문서 업데이트 (필요 시)
- [ ] 코드 영향 없음 확인 (필요 시)

## PR/커밋 규칙
- 작업 범위 명확히 기술
- 관련 SSOT 문서 참조
- 변경 파일 목록 포함
